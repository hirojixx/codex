1. 何についての資料か

理解のために単純化すると、SQLは「表から行を取り出す言語」ではなく、制約付きのデータ構造に対して、宣言的に結果集合を指定し、DBMSのオプティマイザに実行方法を選ばせる言語である。厳密には後述するが、この見方を持つと、文法・設計・性能問題が一つにつながる。

重要な読み方SQL標準で定義されるもの、Oracleで実装・拡張されるもの、運用環境に依存するものを混ぜると誤解しやすい。資料内では SQL標準 Oracle 共通概念 注意 で切り分ける。

2. なぜSQL標準とOracle前提を分けるべきか

SQLはISO/IEC 9075として標準化されているが、実務で使うSQLは常にDBMS実装の影響を受ける。移植性、性能、運用性を判断するには「標準で考える領域」と「Oracleに寄せてよい領域」を分ける必要がある。

観点SQL標準として考えるOracle前提で考える判断ポイント
データ取得SELECT、FROM、WHERE、GROUP BY、HAVING、ORDER BY、JOIN、サブクエリROWNUM、FETCH FIRST、階層問合せCONNECT BY、MODEL、MATCH_RECOGNIZE、JSON/GRAPH/VECTOR関連移植性が必要なら標準寄り。Oracleで長期運用するなら可読性・性能・機能を優先してOracle機能を使う。
データ型数値、文字、日時、真偽値、NULL、標準的な型概念NUMBER、VARCHAR2、DATEが時刻を持つ、TIMESTAMP WITH LOCAL TIME ZONE、CLOB/BLOB、JSON、VECTOR、BOOLEAN型変換・NLS・桁あふれ・比較規則が性能と正確性に直結する。
制約PRIMARY KEY、FOREIGN KEY、UNIQUE、CHECK、NOT NULLDEFERRABLE、RELY/NOVALIDATE、仮想列、関数ベース索引、ドメイン索引との連動アプリで守る制約は漏れやすい。DB制約はデータ品質とオプティマイザの前提にもなる。
性能制御標準SQLは実行方法を基本的に指定しないヒント句、実行計画、統計、SQL Plan Baseline、SQL Profile、初期化パラメータヒントは最後の手段ではなく「限定的な制御手段」。ただし乱用するとアップグレード耐性を下げる。
はまりポイント: “ANSI SQLならどこでも動く” ではない標準SQLは共通語彙を与えるが、実装差は大きい。特に日付時刻、文字照合、NULL、MERGE、再帰問合せ、ロック、分離レベル、DDLの暗黙コミット、ヒント、実行計画はDBMSごとの差が実害になりやすい。

4. SQL標準の主要機能

4.1 SELECTの論理的な処理順序

SQL標準 SQLは宣言的言語なので、実際の物理実行順と論理的な意味の順序は一致しない。理解用には次の順序で考えるとよい。

FROM / JOIN
WHERE
GROUP BY
HAVING
SELECT
DISTINCT
ORDER BY
OFFSET / FETCH
はまりポイント: SELECT句の別名はWHEREで使えないことが多い論理的にはWHEREがSELECTより前に評価されるため、SELECTで付けた別名をWHEREで使えると期待すると移植性が落ちる。OracleでもORDER BYでは別名を使えるが、WHEREでは通常使えない。

4.2 JOIN

SQL標準 JOINは「表を横につなぐ」構文だが、実務では行数の増減を支配する。INNER JOIN、LEFT/RIGHT/FULL OUTER JOIN、CROSS JOIN、自己結合、半結合/反結合相当の書き方を理解する。

種類意味典型用途はまりポイント
INNER JOIN結合条件に合う行だけを残す注文と顧客など必須関係結合キーの重複で行が増える。1対多か多対多かを先に確認。
LEFT OUTER JOIN左表の行を残し、右表がなければNULL任意の関連データを付与右表条件をWHEREに書くと外部結合が内部結合化しやすい。
EXISTS存在確認親に子が存在するかSELECTリストより相関条件が重要。NOT IN + NULLよりNOT EXISTSが安全な場合が多い。
集合演算UNION/INTERSECT/EXCEPT相当結果集合の合成OracleではMINUSが伝統的。重複排除コストに注意し、必要ならUNION ALL。

4.3 集約・ウィンドウ関数

SQL標準 GROUP BYは行を集約して行数を減らす。ウィンドウ関数は行数を維持しながら、パーティション内順位、累計、移動平均などを計算する。

-- 部署ごとの給与合計: 行数は部署数に減る
SELECT dept_id, SUM(salary) AS total_salary
FROM employees
GROUP BY dept_id;

-- 社員行を残しつつ部署内順位を付ける
SELECT emp_id, dept_id, salary,
       RANK() OVER (PARTITION BY dept_id ORDER BY salary DESC) AS dept_rank
FROM employees;
判断基準「グループ単位の1行が欲しい」ならGROUP BY。「明細行を残しながらグループ内情報を足したい」なら分析関数/ウィンドウ関数。

4.4 CTE・再帰・サブクエリ

SQL標準 WITH句は複雑なSQLを分割して読みやすくする。再帰CTEは階層やグラフ探索を表現できる。ただし、Oracleでは従来からCONNECT BYも広く使われ、再帰CTEとの選択が発生する。

はまりポイント: WITH句は必ず中間表を作るわけではないオプティマイザはインライン展開、マテリアライズ、変換を選ぶ場合がある。読みやすさ目的のWITH句が性能上の境界になるとは限らない。

4.5 NULLと三値論理

SQL標準 NULLは「空文字」や「0」ではなく、未知/適用不能を表す。比較結果はTRUE/FALSEだけでなくUNKNOWNを持つ。

-- col = NULL は期待通りに判定できない
WHERE col IS NULL

-- NOT IN は副問合せ側にNULLが混ざると結果が空になり得る
WHERE NOT EXISTS (
  SELECT 1 FROM child c WHERE c.parent_id = p.parent_id
)
重大な注意Oracleでは長年、文字列の空文字 `''` はNULLとして扱われる。この挙動を標準SQLや他DBMSと同じだと思うと、NOT NULL制約、文字列結合、比較、移行で事故が起きる。

4.6 DDL・制約・トランザクション

SQL標準 SQLの理解ではSELECTだけに偏らないことが重要である。表定義、制約、権限、トランザクション境界が、SQLの正しさと性能を決める。

ビューCREATE VIEWSQLの抽象化。権限境界や再利用に便利。ビュー・マージにより実行計画上は展開される場合がある。更新可能ビューの制約に注意。

5. Oracle前提で変わる部分

5.1 データ型

Oracle型/概念理解実務判断はまりポイント
NUMBER高精度な数値型。精度・スケール指定が重要。金額・数量はFLOATではなくNUMBER(p,s)を基本に検討。暗黙変換で索引が効かない、桁あふれ、丸め。
VARCHAR2可変長文字列。Oracle実務の標準的文字列型。VARCHARではなくVARCHAR2を使うのがOracle実務上安全。BYTE/CHARセマンティクス、NLS、空文字NULL。
DATE日付だけでなく時分秒を保持する。日付だけの業務概念でも時刻混入を設計で扱う。`TRUNC(date_col)` をWHEREに書くと索引利用に影響しやすい。
TIMESTAMP WITH TIME ZONEタイムゾーン情報を含む日時。国際化・監査ログ・外部連携で検討。LOCAL TIME ZONEはDB/セッションタイムゾーン変換の理解が必要。
BOOLEANOracle 23ai/26ai系でSQL BOOLEANが利用可能。新規設計では真偽値表現の候補になる。既存システムや他DB移植ではNUMBER(1)/CHAR(1)との変換方針が必要。
JSON / VECTOR半構造データ・AI Vector Search系の用途。リレーショナル中心か、検索/AI機能をDB内で使うかで判断。便利さで正規化を崩すと制約・更新・統計・索引設計が難しくなる。

5.2 Oracle固有/Oracle色の強い構文

  • ROWNUM: 行番号風の疑似列。ORDER BYより前に評価される文脈があり、上位N件ではFETCH FIRSTやサブクエリとの関係を理解する。
  • ROWID: 行の物理的位置に近い識別子。高速アクセスや診断に使えるが、論理キーの代替にしない。
  • CONNECT BY: 伝統的な階層問合せ。再帰WITHと比較して、既存資産・可読性・標準性で選ぶ。
  • MERGE: UPSERT用途で便利。ただし一意性、同一行複数回更新、トリガー、ロック競合を確認する。
  • MODEL / MATCH_RECOGNIZE / PIVOT: 複雑な分析をSQL内で表現できるが、保守できるチームかを判断する。
  • ヒント句: コメント構文 `/*+ ... */` でオプティマイザに指示を与えるOracle固有の実務手段。
Oracle機能を使う判断「移植性より性能・保守性・Oracle運用との親和性が重要」「Oracle Databaseを中長期で使う」「チームが実行計画を読める」ならOracle機能は有力。逆に、将来DB移行が現実的な場合、アプリSQLを標準寄りに保つ選択肢も残す。

6.2 正規化と非正規化

共通概念 正規化は重複と更新異常を減らす。非正規化は読み取り性能や集計容易性のために冗長性をあえて持つ。どちらか一方が常に正しいわけではない。

正規化を優先する場面マスタ・トランザクションの整合性が重要、更新が多い、履歴・監査が必要、データ品質をDB制約で守りたい。
非正規化を検討する場面集計・検索が支配的、応答時間要件が厳しい、集約済みビューやマテリアライズドビューで整合性の許容範囲を設計できる。

6.3 索引設計

Oracle 索引は万能な高速化装置ではない。読み取りを速くする代わりに、INSERT/UPDATE/DELETE、領域、統計、実行計画の複雑性を増やす。

索引候補使う場面避ける/注意する場面Oracleでの観点
B-tree索引選択性の高い検索、結合キー、ORDER BY支援低選択性列単独、更新頻度が高すぎる列複合索引の列順、クラスタリングファクタ、索引スキップスキャン。
ビットマップ索引DWHで低カーディナリティ列を組み合わせるOLTPで高頻度更新ロック競合に注意。DWH向き。
関数ベース索引`UPPER(col)` や `TRUNC(date_col)` 条件を支援関数の決定性やNLS依存が曖昧SQL式と索引式の一致、統計収集が重要。
パーティション索引大規模表、期間分割、保守性小規模表、分割キーとアクセスパターンが合わないローカル/グローバル索引、パーティションプルーニング。
はまりポイント: “索引を張れば速い” ではない件数が多くても全体の大半を読むならフルスキャンが妥当な場合がある。逆に件数が少なくても暗黙変換・関数適用・統計不備で索引が使われないことがある。

7.2 実行計画を見るときの順序

  1. 問題SQLのSQL_IDを特定する。
  2. 実際に実行されたカーソルの計画を `DBMS_XPLAN.DISPLAY_CURSOR` で確認する。
  3. 推定行数(E-Rows)と実績行数(A-Rows)の差を見る。
  4. Predicate Informationで暗黙変換、関数適用、フィルタ条件を確認する。
  5. 待機イベント、I/O、TEMP使用量、PGA、並列度、バッファ取得を確認する。
  6. SQL本文、統計、索引、パーティション、ヒント、パラメータのどこで直すべきかを分ける。
EXPLAIN PLANだけで判断しないEXPLAIN PLANは実行時のバインド、実カーソル、適応計画、統計フィードバック、実行時情報を完全には反映しないことがある。性能調査では実行済みカーソルと実測値を優先する。

8. チューニング手順

チューニングは「SQLを短くする作業」ではない。目標、測定、原因切り分け、変更、検証、固定化の順で進める。

段階やることOracleで見るもの失敗パターン
1. 目標設定何秒以内、何件、同時実行数、許容リソースを決めるAWR/ASH、SQL Monitor、アプリログ単発実行だけ速くし、同時実行で悪化。
2. 測定実行時間、CPU、I/O、待機、行数、TEMPを測るV$SQL、DBMS_XPLAN、Real-Time SQL Monitoringキャッシュが温まった1回だけで判断。
3. 原因分類SQL本文、設計、統計、索引、パラメータ、ロックを分ける実行計画、A-Rows/E-Rows、待機イベントすぐヒントを入れて根本原因を隠す。
4. 改善最小変更で仮説を検証SQL書換、統計、索引、パーティション、ヒント複数変更を同時に入れて効果が不明。
5. 固定化再現性、回帰、運用手順を残すSQL Plan Baseline、テスト、監視閾値本番だけ速い/遅い理由を説明できない。

8.1 SQL本文で直す候補

  • SELECT * を避け、必要列だけ取得する。
  • 結合条件を明示し、意図しない多対多結合を排除する。
  • WHERE列に関数をかける代わりに範囲条件へ変換する。例: `TRUNC(order_date)=DATE '2026-06-02'` ではなく `order_date >= ... AND order_date < ...`。
  • NOT INにNULLが混ざる可能性がある場合はNOT EXISTSを検討する。
  • UNIONの重複排除が不要ならUNION ALLにする。
  • ページングでは安定したORDER BYを持たせる。同値時の順序不定に注意する。
  • 巨大なINリスト、暗黙変換、ワイルドカード前方一致、OR連発は実行計画に注意する。

8.2 設計/統計/索引で直す候補

  • 外部キー列に索引が必要かを、親更新/削除、子検索、ロック影響で判断する。
  • 複合索引は「等価条件」「範囲条件」「ORDER BY/GROUP BY」「選択性」「圧縮」を合わせて列順を決める。
  • 相関列は拡張統計を検討する。例: 都道府県と市区町村、ステータスと完了日。
  • パーティションはアクセス範囲削減だけでなく、データ削除、ロード、統計、索引保守のためにも使う。
  • 統計収集は負荷、鮮度、ヒストグラム、サンプル、保留統計、ロック統計を運用設計に含める。
チューニングの副作用あるSQLを速くする索引やパラメータが、別SQLのDMLコスト、実行計画、メモリ、TEMP、ロックを悪化させることがある。単独SQLだけでなくワークロード全体で判断する。

8.4 SQLレビューのチェックリスト

  • 業務キーとDB制約は一致しているか。
  • WHERE条件の列とバインド型は一致しているか。
  • 結合条件は全て書かれているか。意図しないCROSS JOINはないか。
  • 外部結合の条件がON/WHEREのどちらに置かれているか。
  • GROUP BY前後で行数が想定通りか。
  • ORDER BYはページングやFETCH FIRSTに対して決定的か。
  • 実行計画の推定行数と実績行数に大きな差がないか。
  • 統計・ヒストグラム・拡張統計が必要なデータ偏りがないか。
  • 索引追加のDML影響、領域、保守、重複索引を確認したか。
  • 本番相当のデータ量・偏り・同時実行で測ったか。

9. ヒント句とSQL Plan Management

9.1 ヒント句の位置づけ

Oracle ヒント句はOracle SQLのコメント内に書くオプティマイザへの指示である。標準SQLではない。ヒントは、統計やSQL構造だけでは望ましい計画にできない場合、影響範囲をSQL単位に閉じ込めて制御する手段になる。

SELECT /*+ gather_plan_statistics leading(o c) use_hash(c) */
       o.order_id, c.customer_name
FROM orders o
JOIN customers c ON c.customer_id = o.customer_id
WHERE o.order_date >= DATE '2026-01-01';
カテゴリ代表ヒント使う場面注意
最適化目標ALL_ROWS, FIRST_ROWS(n)全体スループットか初期応答かを明確にしたいアプリのフェッチ方法と一致しないと期待外れになる。
アクセスパスFULL, INDEX, INDEX_FFS, NO_INDEX索引/全表走査の選択を制御したいデータ量変化で逆効果になりやすい。
結合順LEADING, ORDERED駆動表を制御したいクエリブロック名を使わないと意図しない箇所に効く/効かない。
結合方式USE_NL, USE_HASH, USE_MERGE小規模駆動表、巨大結合、ソート済み結合などを制御メモリ不足、TEMP使用、同時実行で悪化しうる。
SQL変換NO_MERGE, MERGE, UNNEST, NO_UNNEST, MATERIALIZE, INLINEビュー/サブクエリ/CTEの変換を制御Oracleバージョン差、クエリブロック指定、保守性に注意。
並列PARALLEL, NO_PARALLEL, PQ_DISTRIBUTE大規模スキャン/集計/ロードリソースを食い尽くしやすい。Resource Managerや並列設定と併せる。
診断GATHER_PLAN_STATISTICS, MONITOR実行時統計を取りたい診断目的。恒久的に残す場合は負荷と情報漏えいに注意。

9.2 ヒントが効かない/危ない理由

  • 構文上はコメントなので、タイプミスしてもエラーにならず無視される場合がある。
  • 別名が一致していない、クエリブロックが違う、ビュー変換後に対象が変わる。
  • 矛盾するヒント、実行不可能なアクセスパス、制約や権限で成立しない指定は採用されない。
  • データ量・統計・バージョンが変わると、固定した計画が古くなる。
  • ヒントで速くなったように見えて、根本原因が統計不備や設計不備の場合がある。

9.3 SQL Plan Management / SQL Profile / SQL Patch

機能目的向く場面注意
SQL Plan Baseline既知/検証済み計画を使わせ、計画回帰を抑えるアップグレード、統計更新、重要SQLの安定化悪い計画を固定しない。進化/evolve運用が必要。
SQL Profile見積り誤差を補正する補助情報を与えるSQL Tuning Advisorの推奨、カーディナリティ推定ずれ計画を直接固定するものではない。
SQL PatchSQL本文を変えずにヒント等を適用アプリ改修が難しい緊急対応隠れた変更になりやすいので台帳化・期限設定が必要。
ヒント埋め込みSQL単位で明示的に制御SQLを所有し、レビュー・テストできる場合移植性低下、将来のデータ変化への追随が弱くなる。
実務方針まず統計・設計・SQL本文を疑う。次に、変更できるSQLなら限定的なヒント。変更できない/重要SQLの安定化ならSPMやPatch。いずれも「なぜ必要か」「いつ外すか」を記録する。

10. DB設定・初期化パラメータ

Oracle DB設定はSQL性能に影響するが、パラメータ変更は広範囲の副作用を持つ。SQL単体の問題をインスタンス全体設定で直そうとすると、別ワークロードを壊すことがある。

12. 例で理解する

12.1 最小例: 日付条件

-- 悪い例: 列に関数をかけるため、通常の索引が使いにくい
WHERE TRUNC(order_date) = DATE '2026-06-02'

-- 改善例: 半開区間で書く
WHERE order_date >= DATE '2026-06-02'
  AND order_date <  DATE '2026-06-03'

見るべき点は「同じ意味に見える条件でも、列側に関数をかけるかどうかでアクセスパスが変わる」ことである。Oracle DATEは時刻を持つため、等価比較だけでは当日全体を表せない。

12.2 実務例: 売上集計SQLを段階分割する

WITH target_orders AS (
  SELECT order_id, customer_id, order_date
  FROM orders
  WHERE order_date >= DATE '2026-01-01'
    AND order_date <  DATE '2026-02-01'
), order_amounts AS (
  SELECT oi.order_id, SUM(oi.quantity * oi.unit_price) AS amount
  FROM order_items oi
  JOIN target_orders o ON o.order_id = oi.order_id
  GROUP BY oi.order_id
)
SELECT c.customer_id, c.customer_name, SUM(a.amount) AS total_amount
FROM order_amounts a
JOIN target_orders o ON o.order_id = a.order_id
JOIN customers c ON c.customer_id = o.customer_id
GROUP BY c.customer_id, c.customer_name
ORDER BY total_amount DESC
FETCH FIRST 20 ROWS ONLY;

この例では、対象期間の注文を先に論理的に切り出し、明細集約、顧客集約の順に意図を分けている。実際に中間表が作られるとは限らないため、必要なら実行計画でWITH句の扱いを確認する。

12.3 失敗例: ヒントで根本原因を隠す

-- とりあえず索引を強制した例
SELECT /*+ INDEX(o orders_status_idx) */ *
FROM orders o
WHERE status = 'CLOSED'
  AND order_date >= DATE '2026-01-01';

status='CLOSED' が全体の大半なら索引強制は逆効果になりうる。根本原因は、複合索引の列順、ヒストグラム、期間条件、パーティション、あるいは集計設計かもしれない。ヒントを入れる前に、A-Rows/E-Rows、Predicate Information、I/O、TEMP、待機を確認する。
