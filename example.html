1テーブル設計の責務 ― 何を守るための設計か

テーブル設計は、画面項目を保存する箱を作る作業ではない。上級者は、データの意味・同一性・履歴・整合性・変更容易性を、将来の業務変更と性能要件に耐える形へ配置している。

テーブル設計で守るべき6つの責務

責務設計で決めること雑に扱ったときの地獄
意味の固定「申請」「会員」「ユーザー」「アカウント」などの境界を決める同名異義が1表に混ざり、NULLだらけ・条件分岐だらけのゴッドテーブルになる
同一性の固定内部ID、業務キー、外部ID、表示番号を分ける外部システムのキー変更で全FKが崩れる。名寄せ不能。過去データの意味が変わる
時間の固定現在値・履歴・有効期間・発生時刻・登録時刻を分ける「いつそうだったか」「なぜそうなったか」を後から説明できない
整合性の固定PK/FK/UNIQUE/CHECK/NOT NULL/トランザクション境界を決める壊れたデータが入った後でアプリ側だけでは修復不能になる
検索・集計の固定どの画面・帳票・APIがどの形で読むかを意識する正規化は美しいが毎回重いJOINになり、後から場当たり的な列追加が始まる
進化余地の固定後で型・意味・状態・外部連携が変わる前提で逃げ道を作る列削除・キー変更・履歴追加が大工事になる。データが失われた後で気づく
機序: テーブルは「仕様」になる
一度本番データが入ると、テーブル定義はコードよりも硬い仕様になる。アプリは置き換えられるが、蓄積済みデータの意味は簡単には変えられない。だから設計時点で「何を事実として残すか」「何は導出でよいか」「何を消してはいけないか」を分ける必要がある。

画面項目から始めると失敗する理由

画面は業務の入口だが、データの寿命とは一致しない。画面上は1つの「ステータス」に見えても、DB上では現在状態、状態遷移履歴、承認タスク、通知ログ、監査証跡に分かれることがある。逆に、画面上は複数の入力欄でも、DB上では1つの値オブジェクトやコード体系にまとめた方が安全なこともある。

実務では、まず「将来説明責任が発生する事実」を特定する。申請が却下された事実、誰が差し戻したか、外部IDがいつ変わったか、部品の構成がどの版で有効だったかは、画面から消えても業務上は残る。この種の事実を現在値列の上書きだけで扱うと、後から監査・復元・調定ができなくなる。

2全体マップ ― テーブル設計の判断順序

理解のために単純化すると、DB設計は「何を表すか」から「どう速く読むか」へ流れる。逆に、最初から索引や列名だけ考えると、意味の境界を誤ったまま高速な負債を作る。

  1. 概念を切る：業務用語を列挙し、同名異義・異名同義を潰す。例: ユーザー、会員、申請者、承認者、アカウントは同じとは限らない。
  2. リソース・イベント・状態を分ける：在り続けるもの、起きたこと、現在の状態、状態遷移履歴を混ぜない。
  3. 同一性を決める：代理キー、自然キー、外部ID、表示番号、版番号を分ける。変更される可能性のあるキーをPKにしない。
  4. 制約を決める：DBで守る不変条件、アプリで守る遷移条件、バッチで調定する条件を分ける。
  5. 時間を決める：発生時刻、登録時刻、有効期間、処理時刻、削除時刻を分ける。どれを履歴化するか明示する。
  6. 読み方を決める：正規化した書込モデルに対し、一覧・検索・帳票のためのビュー、MV、検索テーブルを検討する。
  7. 変え方を決める：本番稼働後に列追加・分割・型変更・外部ID変更が起きる前提で、移行手順を設計に含める。
実務のレビュー観点
テーブル一覧を見たときに「これは何の事実を保存している表か」「この行はいつからいつまで有効か」「この行を削除してよいか」「このIDは誰が発行し、変更されるか」を説明できない表は危険である。

3業務概念の切り分け ― 同じ言葉を同じ表にしない

テーブル設計の事故は、列の型よりも概念の境界で起きる。「顧客」「ユーザー」「会員」「申請」「部品」のような言葉は、部門・画面・外部システムによって意味がずれる。

同名異義・異名同義を先に処理する

現象例設計上の扱い
同名異義営業の「顧客」は見込み客を含むが、請求の「顧客」は請求先だけ1表に統合せず、文脈ごとの集約を分ける。共有するのは同一性または参照だけ
異名同義品目・部品・アイテム・SKUが同じ実体を指している用語を統一し、別表にしない。別名はコード体系・表示名として扱う
状態違い入会申請者と会員は同じ人物だが、業務上の責務が違うPerson/User と MembershipApplication / Membership を分ける
役割違い同じユーザーが申請者にも承認者にもなるユーザー表に「承認者フラグ」を固定せず、ロール・権限・組織所属で表す
版違い部品AのRev.1とRev.2は同じ部品か別物か設計・購買・製造のどこで同一視するかにより、part と part_revision を分ける

概念の境界は、最終的に不変条件の境界でもある。たとえば「申請」は差し戻し・却下を持つが、「会員」は入会日・退会日・会員番号を持つ。これを1つの users 表に詰め込むと、申請中ユーザーには会員番号がNULL、退会済ユーザーにはログイン不可フラグ、却下ユーザーには却下理由、招待中ユーザーには期限、というように、1表の中に複数のライフサイクルが混ざる。

ゴッドテーブル化の初期症状
列名に xxx_type、xxx_status、xxx_flag、xxx_date1、memo1 が増え、NOT NULLにできる列が少なくなり、CHECK制約がほとんど書けない。この状態は、概念が1表に混ざっている可能性が高い。

境界を切るための質問

  • その行は、業務上いつ生まれ、いつ消えるのか。
  • その行の主語は誰か。人か、アカウントか、会員資格か、申請か、承認タスクか。
  • 同じIDで追跡したい寿命はどこまでか。申請から退会までか、再入会後も同一か。
  • その表の全行に必ず成り立つ不変条件を1文で言えるか。
  • 同じ言葉でも、外部システムや部門で意味が違わないか。

4リソース・イベント・状態 ― まずこの3つを分ける

既存資料でも強調されていた通り、リソースは「在り続けるもの」、イベントは「起きたこと」、状態は「イベントの結果として現在どうなっているか」である。この3つを混ぜると履歴と監査が壊れる。

種類性質典型テーブル主な操作失敗例
リソース同一性を持ち、属性が変わるmember, part, supplier, accountINSERT / UPDATE部品名変更で過去の帳票まで新名称に見える
イベント発生時刻を持ち、原則不変membership_event, approval_event, inventory_movementINSERT only承認済フラグだけで、誰がいつ承認したか分からない
状態リソースの現在の局面application.current_status, membership.status現在値更新＋履歴追記status上書きだけで差戻履歴が消える
射影検索・一覧・集計のための導出形member_search_view, application_summary_mv再生成可能な更新射影を真実扱いして、再計算不能になる

イベントを可変列で持つと何が消えるか

例として membership.status = 'APPROVED' だけを持つと、「承認された」という現在状態は分かる。しかし、誰が承認したか、承認前に差し戻しがあったか、却下後に再申請されたか、システム処理で承認されたか、人が判断したかは消える。状態列は高速な検索には便利だが、事実の説明責任を持てない。

上級者は、状態を「現在値」として持つ場合でも、状態を変えたイベントまたは履歴を別表に残す。現在値は読みを速くするためのキャッシュ、履歴は説明責任を持つ事実、と役割を分ける。

基本形
基幹系では「現在値列 + 状態遷移履歴 + 必要ならイベントログ」が基本形になる。現在値だけは速いが消える情報が多く、イベントだけは正確だが現在値検索が重くなる。二段持ちの冗長性は、意図的な妥協である。

5キーと同一性 ― 主キー・業務キー・外部IDを混ぜない

主キーは「行を不変に識別するための内部的な錨」である。業務上の番号、外部システムのID、表示用コード、版番号は重要だが、変更される可能性があるなら主キーにしない。

キーの種類

キー意味変更可能性推奨用途
代理キー / surrogate keyDB内部で発行する不変ID。例: member_id原則変更しないPK、FKの参照先。長期保存データの同一性
自然キー / natural key業務上の一意識別子。例: 会員番号、品番、法人番号制度・連携・業務都合で変わることがあるUNIQUE候補。ただし変更・再利用・欠損を確認
外部ID外部システムが発行するID。例: CRM顧客ID、PLM部品ID相手都合で変わる。重複・再利用もありうる対応表で管理。PK/FKに直結しない
表示番号画面や帳票に出す番号。例: 申請番号 APP-2026-0001採番ルール変更・再発番がありうるUNIQUE制約は可。ただし内部IDとは分ける
版キーRev、版、適用開始日など変更というより新しい版が増えるpart_revision など別表化し、親ID + revision_no を一意にする

変更される可能性がある主キーの扱い

外部システム連携がある場合、「外部から来るIDが一意だからPKにする」は危険である。外部IDが後から訂正された場合、自システム内のFK、履歴、監査ログ、メッセージ、帳票がすべて古い値を参照する。技術的にはON UPDATE CASCADEを持つDBもあるが、Oracleでは主キーをFKで広く参照させてから値を変える設計自体を避ける方が安全である。

実務解は、内部代理キーを不変の主キーにし、外部IDは external_identifier 対応表で持つ。対応表には、外部システム名、外部ID、内部ID、有効期間、取り込みバッチ、確信度、訂正理由を持たせる。これにより、外部ID変更は「対応表の履歴追加」として扱え、内部の参照は壊れない。

CREATE TABLE member (
  member_id       NUMBER GENERATED ALWAYS AS IDENTITY,
  member_no       VARCHAR2(40),
  display_name    VARCHAR2(200) NOT NULL,
  created_at      TIMESTAMP DEFAULT SYSTIMESTAMP NOT NULL,
  CONSTRAINT pk_member PRIMARY KEY (member_id),
  CONSTRAINT uq_member_no UNIQUE (member_no)
);

CREATE TABLE external_identifier (
  external_identifier_id NUMBER GENERATED ALWAYS AS IDENTITY,
  entity_type      VARCHAR2(40) NOT NULL,
  entity_id        NUMBER NOT NULL,
  system_code      VARCHAR2(40) NOT NULL,
  external_key     VARCHAR2(200) NOT NULL,
  valid_from       TIMESTAMP DEFAULT SYSTIMESTAMP NOT NULL,
  valid_to         TIMESTAMP DEFAULT TIMESTAMP '9999-12-31 00:00:00' NOT NULL,
  source_batch_id  VARCHAR2(80),
  confidence_score NUMBER(5,2),
  CONSTRAINT pk_external_identifier PRIMARY KEY (external_identifier_id),
  CONSTRAINT uq_extid_current UNIQUE (entity_type, system_code, external_key, valid_to)
);
注意: 外部ID対応表のFK
上記例は entity_type + entity_id の汎用対応表であり、そのままでは entity_id に厳密なFKを張りにくい。厳密性を優先するならエンティティごとに対応表を分けるか、共通の entity_registry を親にする。汎用性と参照整合性はトレードオフになる。

6制約と不変条件 ― DBで守るもの、アプリで守るもの

制約は、アプリのバグやバッチの事故からデータを守る最後の防壁である。OracleではPK、UNIQUE、FK、CHECK、NOT NULL、DEFERRABLE、NOVALIDATEなどを使い分ける。

制約の使い分け

制約守れること守れないこと設計上の注意
PRIMARY KEY行の同一性。NULL不可、一意業務上の意味の正しさ変更される値をPKにしない。複合PKはFK側が重くなる
UNIQUE候補キー、重複禁止NULLを含む場合の意図しない複数行条件付き一意は関数索引・仮想列で表現
FOREIGN KEY参照先の存在参照先の状態条件。例: 承認済の親だけ参照親子が同一DB内にあることが基本。分散は別途対策
CHECK1行内で完結する条件他行・前状態・集計条件状態遷移ルールや期間重複はCHECKだけでは守れない
NOT NULL必須項目業務的に「いつ必須か」が条件付きの項目状態により必須なら表分割やCHECKを検討
DEFERRABLEトランザクション末尾まで制約検査を遅延すべての制約を万能に遅延できるわけではない一括入替・循環参照・スナップショット更新で有効

Oracleの外部キーは、子の値が親の主キーまたは一意キーに一致することを要求する。複合外部キーは、親側の複合PK/UNIQUEと列数・型が対応している必要がある。これは参照整合性の基本であり、設計レビューでは「FKを張れる形になっているか」を最初に見る。

条件付き一意制約の例

退会済や論理削除済を残しつつ、アクティブな会員だけメールアドレスを一意にしたい場合、単純な UNIQUE(email) では過去行と衝突する。Oracleでは仮想列または関数ベース一意索引を使う。NULLは一意索引上で複数許容されるため、条件外をNULLに落とすのが定番である。

CREATE TABLE member_account (
  member_account_id NUMBER GENERATED ALWAYS AS IDENTITY,
  email             VARCHAR2(320) NOT NULL,
  closed_at         TIMESTAMP,
  active_email_key  VARCHAR2(320)
    GENERATED ALWAYS AS (
      CASE WHEN closed_at IS NULL THEN LOWER(email) END

機序: 制約が効く範囲を意識する

制約を後付けするとき

既存データが大量にある本番では、いきなり全行検査付きで制約を追加すると長時間ロックや失敗が起きる。Oracleでは ENABLE NOVALIDATE により「既存行は検証しないが、新規DMLには効かせる」という段階を作れる。移行では、まず新規の破壊を止め、既存不整合を別途調定し、最後にVALIDATEする。

ALTER TABLE approval_request ADD CONSTRAINT fk_approval_request_member
  FOREIGN KEY (member_id) REFERENCES member(member_id)
  ENABLE NOVALIDATE;

-- 既存不整合を調査して修正後、検証へ進める
ALTER TABLE approval_request MODIFY CONSTRAINT fk_approval_request_member VALIDATE;

11時間・履歴・時系列 ― 発生時刻、有効期間、記録時刻を分ける

履歴設計で最も重要なのは、時間の種類を混ぜないこと。業務上いつ有効だったか、いつ発生したか、いつDBへ記録されたか、いつ訂正されたかは別の意味を持つ。

時間の種類

時間意味例典型列
発生時刻 / event time業務上の出来事が起きた時刻承認した時刻、入庫した時刻occurred_at, approved_at

時系列データは大きく3つに分かれる。業務的な低頻度履歴、トランザクションイベント、高頻度計測値である。RDBが得意なのは前2つであり、高頻度センサやログを基幹RDBへ無差別に入れると、索引・パーティション・保持期間・集計で苦しくなる。

トランザクション履歴入出庫、承認、残高変動イベント追記 + 必要なら現在値/集計を別途キャッシュ
高頻度時系列IoT、監視メトリクス、アプリログ専用ストアまたはパーティション表。RDBには集約済みや業務判断に必要な粒度のみ

半開区間の基本

期間履歴は、下端を含み上端を含まない [valid_from, valid_to) で持つと扱いやすい。2026-04-01から2026-05-01まで有効なら、条件は valid_from <= t AND t < valid_to になる。隣接する期間の境界が重複しないため、日付・時刻の精度変更にも強い。

13ユニークキーが存在しないデータ ― 無理に一意化しない

現実のデータには、ユニークキーが存在しない、欠損している、重複する、誤っている、後から判明するものがある。無理に一意制約を貼ると、取り込み不能・誤名寄せ・データ消失が起きる。

一意でないデータの扱い

状況例やってはいけないこと設計
完全な一意キーがない氏名・生年月日・電話番号だけ氏名+電話をPKにするraw取り込み表 + 内部ID + 名寄せ候補表
重複が意味を持つ同じ品番が取引先ごとに違う品番単独UNIQUEにするスコープ付き一意。例: supplier_id + supplier_part_no
欠損がある外部IDが一部NULLNULLをダミー値で埋める未確定IDとして扱い、後で対応表に追加
誤入力があるメールアドレス重複、法人番号誤り古い行を上書きして消す訂正履歴・検証状態を持つ
同一性が確率的会社名表記揺れ自動で1社に確定統合する候補・確信度・人手確認ステータスを持つ

Raw / Canonical / Match の三層

外部ファイル・スクレイピング・手入力・古い基幹データのように品質が低いデータは、正規表へ直接入れない。まず raw 表に原文を保存し、canonical 表へ正規化し、match 表で「どの内部エンティティ候補に対応するか」を管理する。これにより、後から名寄せロジックを改善しても原文を失わない。

危険: 取り込み時に重複を捨てる
MERGE や INSERT IGNORE 的な処理で、重複と見なした行を捨てると、後から誤名寄せに気づいても復元できない。生データは必ず raw として保持し、正規化・名寄せは再実行可能にする。

一意性のスコープを見つける

「ユニークキーがない」と思えるデータでも、スコープを足すと一意になることがある。取引先品番は取引先内では一意、会員番号は会員制度内では一意、部品Revは部品内では一意、申請番号は年度・組織内では一意、という形である。一意制約を貼る前に、必ずスコープを言語化する。

14データ削除と消失 ― 物理削除してよいもの、いけないもの

「不要になったからDELETE」は、多くの業務データでは危険である。削除には、物理削除、論理削除、終了、有効期限切れ、取消、無効化、訂正があり、それぞれ意味が違う。

削除の種類

種類意味例設計
物理削除DBから行を消す一時作業データ、再生成可能キャッシュ監査不要・参照なし・再生成可能なものに限定
論理削除削除済みフラグを立てるユーザーが削除した下書き削除者・削除時刻・理由を持つ。UNIQUE条件に注意
終了ライフサイクルが終わった退会、契約終了statusやvalid_toで表す。過去事実は残す
取消起きた事実を取り消す事実申請取消、仕訳取消取消イベントを追加。元イベントは消さない
訂正誤りを正す承認者訂正、金額訂正旧版を閉じて新版を追加。訂正理由を残す
匿名化個人情報を消すが取引事実は残す退会後の個人情報削除個人識別情報と業務事実を分離して設計
データが消えてなくなる典型
状態を上書きする、履歴を持たない、外部取り込みのrawを捨てる、削除済み行を物理削除する、親削除で子をCASCADE削除する、番号変更で過去FKを更新する。これらは、後から「過去を説明したい」となった時点で取り返しがつかない。

退会は削除ではない

会員の退会は、会員資格の終了であって、人物・アカウント・過去申請・承認履歴・請求履歴を消すこととは別である。個人情報保護のために匿名化や削除が必要な場合でも、業務事実を保持するテーブルと個人識別情報を持つテーブルを分けておくと、保持と削除の両立がしやすい。

BOMの現在値だけ管理する危険
現在BOMだけを上書きすると、過去に出荷した製品の構成、原価計算、品質問題の影響範囲、部品交換履歴が再現できない。製造・PLMでは「いつの、どの版の構成か」が本質的なデータである。

サイクル検出

BOMは通常DAGであり、親部品の子に自分自身や祖先が入ると展開が壊れる。単純な parent <> child CHECKでは直接自己参照しか防げない。深いサイクル検出は再帰クエリ、トリガ、アプリの登録処理、または閉包表で守る。

閉包表の検討

多段BOMを頻繁に展開するなら、親子エッジ表だけでは再帰コストが重くなる。祖先・子孫・深さを持つ閉包表を維持すると読取は速いが、BOM変更時の更新が重くなる。書込頻度が低く読取頻度が高いPLM/原価/影響分析では検討価値がある。

第 IV 部 性能と進化 ― 正しさを壊さずに速くし、変えられる設計にする

20正規化と非正規化 ― いつ崩すか

正規化は、更新異常と重複を避けるための基本である。しかし、実務では検索・帳票・集計のために非正規化や導出値の保持が必要になる。重要なのは、真実の源泉とキャッシュを混ぜないこと。

非正規化の判断表

状況正規化のまま非正規化を検討守る条件
通常の業務更新更新頻度が普通でJOINも軽い不要制約で整合性を守る
一覧画面が重い索引・JOIN見直しで足りる検索ビュー、MV、検索テーブル元データから再生成できる
集計が重い都度集計で許容日次/月次集計テーブル更新タイミングと遅延を明記
現在値取得が多い履歴最新行を毎回取る現在値列を持つ履歴更新と同一Txで同期
外部API向け正規表からレスポンス生成API専用読取モデル書込モデルを汚さない
機序: 書込モデルと読取モデルの責務が違う
書込モデルは整合性・更新容易性・不変条件を守るために正規化する。読取モデルは画面・検索・帳票の要求に合わせて冗長でもよい。問題は、読取モデルを唯一の真実として更新し始めることである。

導出値を持つ場合のルール

  • 導出元を明確にする。例: 残高は入出金イベントの合計。
  • 導出値を直接手修正しない。訂正イベントを入れて再計算する。
  • 導出元と導出値の差分を検出する調定クエリを持つ。
  • リアルタイム整合が必要か、数分/日次遅延を許すかを明記する。
  • 再生成できない導出値は、実は導出値ではなく業務事実である可能性を疑う。
-- イベントから現在値を調定する考え方の例
SELECT m.member_id,
       m.cached_point_balance,
       COALESCE(SUM(e.point_delta), 0) AS calculated_balance
FROM member m
LEFT JOIN member_point_event e ON e.member_id = m.member_id
GROUP BY m.member_id, m.cached_point_balance
HAVING m.cached_point_balance <> COALESCE(SUM(e.point_delta), 0);

21ビュー・マテリアライズドビュー・CQRS ― 読む形を分ける

書込モデルを正規化した結果、読みが重くなることはある。その場合、ビュー、マテリアライズドビュー、検索テーブル、CQRS風の読取モデルを段階的に検討する。

選択肢

Oracleのマテリアライズドビューは強力だが、リフレッシュ待ちにより基底データと同期していない状態がありうる。したがって、MVを使う場合は、鮮度要件、リフレッシュ方式、許容遅延、障害時の再構築手順を仕様に含める。

CQRSを局所採用する

CQRSは全面的に導入すると複雑だが、局所採用は有効である。たとえば承認一覧、部品検索、会員ダッシュボード、BOM展開結果など、読みの形が書込の正規形と大きく異なる領域では、書込モデルを守ったまま読取モデルを作る。

CQRSは万能高速化ではない
CQRSは「読みと書きのモデルを分ける」設計であり、単なるキャッシュではない。同期遅延、イベント順序、再実行、重複処理、障害復旧、読取モデル再構築を扱う必要がある。多くのCRUD画面に全面適用すると、複雑性だけが増える。

イベントソーシングは、現在状態を直接保存する代わりに、状態変化イベントの列を真実の源泉にする設計である。履歴・監査・再構築に強いが、実装と運用の難度は高い。

物理設計は最後だが、最後まで無視してよいわけではない。履歴表、イベント表、BOM、承認一覧、外部ID対応表は、典型アクセスがはっきりしているため、設計時点で索引とパーティションを検討する。

最新行取得の落とし穴

履歴テーブルから最新行を取る処理は頻出する。MAX(changed_at) だけでJOINすると、同時刻が複数ある場合に重複する。必ずタイブレーク用の単調増加IDを含めるか、現在値列・現在行フラグ・MVを持つ。

-- 最新状態を安全に取る例: ROW_NUMBERでタイブレークする
SELECT *
FROM (
  SELECT h.*,
         ROW_NUMBER() OVER (
           PARTITION BY h.application_id
           ORDER BY h.changed_at DESC, h.status_history_id DESC
         ) AS rn
  FROM application_status_history h
)
WHERE rn = 1;

パーティション

イベント表や履歴表は時間とともに増え続ける。保持期間、アーカイブ、削除、集計の単位が月/日なら、発生時刻または登録時刻でレンジ/インターバルパーティションを検討する。Oracleのパーティションは可用性・管理性・性能に効くが、キー選択を誤ると剪定が効かない。

索引は読みの前払い・書きの後払い
索引は検索を速くするが、INSERT/UPDATE/DELETEのたびに保守される。承認イベントや入出庫イベントのような追記表では、必要な索引を絞る。検索画面ごとに無秩序に索引を増やす前に、読取モデルやMVの方が適切かを検討する。

Invisible Indexで検証する

OracleのInvisible Indexは、DMLでは保守されるが通常はオプティマイザに使われないため、索引削除や一時索引の検証に使える。既存本番で索引をいきなり削除する前に、見えなくして影響を見る設計が可能である。

同じ業務でも、現在値を1行で更新する設計と、イベントを追記する設計では競合の起き方が変わる。テーブル設計は、同時実行時の正しさにも影響する。

競合が頻繁で処理時間が短い場合、先に対象行をロックする方が分かりやすい。Oracleでは SELECT ... FOR UPDATE などで取得したロックは、コミットまたはロールバックで解放される。複数行をロックする場合は、全処理で同じ順序にし、デッドロックを避ける。

 FOR UPDATE;

25スキーマ進化 ― 後で変更できるテーブルにする

設計は一度では終わらない。列追加、型変更、表分割、状態追加、外部ID追加、履歴追加は必ず起きる。後で変えられる形にしておくこともテーブル設計の一部である。

変更に強い設計原則

原則理由例
意味の再利用をしない同じ列を別意味で使うと移行不能status に承認状態と会員状態を混ぜない
代理キーを使う外部ID変更に耐えるmember_id と member_no を分ける

古い形と新しい形を共存させる無停止移行に必要expand/contract、二重書き、ビュー互換
読取互換をビューで作るアプリ変更を段階化できる旧列名ビュー、統合ビュー

expand / contract の適用

  1. Expand: 新列・新表・新制約を互換的に追加する。既存アプリを壊さない。
  2. Dual Write: 新規更新を旧新両方に書く。先に新規の取りこぼしを止める。
  3. Backfill: 既存データを小さな単位で埋める。冪等にする。
  4. Switch Read: 読みを新形式へ切り替える。差分調定を行う。
  5. Contract: 旧列・旧表・旧処理を削除する。ここで初めて不可逆にする。

Oracleには、DBMS_REDEFINITION によるオンライン再定義、Edition-Based Redefinition による稼働中アップグレード、制約のNOVALIDATE/VALIDATE、Invisible Indexなど、移行を段階化する機能がある。ただし、機能があっても設計が破壊的なら安全にはならない。アプリとDBの互換期間を設計する必要がある。

状態値の追加もスキーマ変更である
列を増やさなくても、status に新しい値を追加するだけで、画面、バッチ、集計、通知、権限、遷移チェック、外部連携が影響を受ける。状態値はDB制約・アプリ列挙型・テストケース・帳票条件と一体で管理する。

第 V 部 レビュー ― 事故を防ぐための確認表とテンプレート

補Aコードマスタ・列挙値・分類体系

ステータス、区分、タイプ、理由コードは、テーブル設計上の小物に見えるが、実際には業務ルールの入口である。CHECK制約で固定するのか、コードマスタで運用変更可能にするのか、階層分類として持つのかを誤ると、後からDDL・画面・バッチ・外部連携が広範囲に壊れる。

CHECK制約かコードマスタか

対象CHECK制約向きコードマスタ向き判断

補B監査列・トレーサビリティ・調定

監査列は、すべての表に機械的に created_at と updated_at を置けばよいわけではない。誰が、どの処理で、どの外部メッセージを根拠に、何を変えたかを追跡できる粒度を決める必要がある。

監査の粒度

粒度例向く用途注意

状態履歴from_status, to_status, changed_byライフサイクル監査状態以外の属性変更は別途必要
イベントログevent_type, payload, actor, occurred_at業務操作の説明イベント設計とバージョン管理が必要
変更前後差分before_value, after_value項目単位の監査量が増える。機密情報の扱いに注意
外部連携ログmessage_id, payload, result再送・冪等性・障害調査raw保持期間と個人情報に注意

調定クエリを設計成果物に含める

履歴と現在値、外部IDと内部ID、イベントと集計値、送信済みメッセージと相手先受信結果など、二重管理や非同期処理を採用した箇所には必ず差分が生じうる。上級設計では、差分をゼロにする処理だけでなく、差分を検出する調定クエリを持つ。

-- 例: 申請の現在状態と最新履歴の差分検出
WITH latest_history AS (
  SELECT application_id, to_status,
         ROW_NUMBER() OVER (
           PARTITION BY application_id
           ORDER BY changed_at DESC, status_history_id DESC
         ) rn
  FROM application_status_history
)
SELECT a.application_id, a.current_status, h.to_status AS latest_history_status
FROM membership_application a
JOIN latest_history h
  ON h.application_id = a.application_id
 AND h.rn = 1
WHERE a.current_status <> h.to_status;
監査は後付けしにくい
監査列は後から追加できるが、過去の「誰が・なぜ」は復元できない。特に承認、退会、削除、価格、BOM、外部連携、権限変更は初期設計で監査粒度を決める。

31状態の持ち方バリエーション総覧 ― status列だけでも、状態テーブルだけでも足りない

状態管理には万能解がない。状態列で持つ、状態履歴を持つ、状態ごとにテーブルを分けてスライドする、イベントから導出する、読取用に現在状態だけ投影する、という複数の形がある。重要なのは「どの方式が正しいか」ではなく、どの不便を受け入れるかを明示することである。

状態を保持する主な方式

方式形強い点弱い点おすすめ度
単一表 + status列users(status)検索・集計・JOINが簡単。小規模CRUDに強い。状態ごとに必須列が違うとNULLだらけ。遷移履歴が消える。単純業務なら可
単一表 + status列 + 状態履歴users + user_status_history現在値の読みやすさと監査を両立。現在値と履歴の二重管理。更新経路統制が必要。基幹系の第一候補

中核エンティティ + 状態別詳細表user_core + active_membership + withdrawal_case同一性を保ちつつ、状態固有データを分離できる。JOINが増える。どの詳細表が現在有効かのルールが必要。おすすめ

キュー/ワークフローエンジンが状態を持つDBは業務実体、状態は外部WF複雑な承認ルート・期限・再割当をエンジンに任せられる。DB単独で全状態を説明しづらい。外部製品依存。大規模WFなら候補
機序: 状態は「分類」ではなく「制約集合」
状態を単なる区分値として見ると status 列で十分に見える。しかし実務上の状態は「その状態で許される操作」「必須になる列」「参照してよい相手」「次に進める状態」「保存すべき履歴」の集合である。つまり状態をどう持つかは、列の配置ではなく、不変条件をどこに置くかの問題である。

再入会・取消・復活があるか状態列だけではなく期間・イベント・ケース化を検討退会を終端状態として単純化できる

32ユーザー退会・停止・再入会の設計 ― 「退会済みフラグ」だけでは足りない

ユーザーの退会管理は、見た目より難しい。退会は単なる status='withdrawn' ではなく、会員資格の終了、ログイン停止、個人情報削除、請求停止、利用規約上の保持、再入会可否、監査証跡、外部連携停止が絡む。これらを1つの状態に押し込むと、後で必ず矛盾が出る。

まず分けるべき概念

方式A: users表に退会状態を持つ

退会管理の追加論点

論点設計上の問い典型的な落とし穴
再入会同じ会員番号を再利用するか。退会前履歴と通算するか。退会済み行を上書きして復活させ、過去の退会事実が消える
退会取消退会申請の取消なのか、退会完了後の復活なのか。取消と再入会を同じ ACTIVE 復帰で処理し、監査不能になる
個人情報削除識別子、連絡先、請求履歴、監査ログをどこまで匿名化するか。FK整合性を壊す物理削除。請求・監査の根拠まで消す
BANと退会BANされた人が退会できるか。再登録を防ぐか。BANを退会と同じ終端状態にして、再登録防止情報を失う
外部連携外部IDを停止するだけか、削除APIを呼ぶか、再試行するか。DBは退会済みだが外部サービスは有効のまま残る
課金退会日、請求締日、返金、未払をどう扱うか。会員状態だけで課金停止し、精算イベントを残さない

退会済みデータをどう残すか

退会後のデータ保持には、少なくとも4つの選択肢がある。

  1. 論理削除: withdrawn_at や deleted_at を持つ。業務検索から除外しやすいが、全SQLで条件を忘れるリスクがある。
  2. アーカイブ表へ移動: 現役表を軽くできるが、FKや横断検索が複雑になる。退会者を完全に別扱いする業務に向く。
  3. 匿名化: 監査・集計に必要な骨格を残し、氏名・メール等を消す。個人情報保護と履歴保持の折衷。
  4. 物理削除: 本当に再現不要な一時データだけ。基幹データでは最後の選択肢。
警告
「退会したから物理削除」は危険。請求・問い合わせ・監査・不正検知・再登録防止の根拠が消える。削除要件がある場合でも、多くは物理削除ではなく匿名化・トークン化・保持期限切れ削除の組合せになる。

33状態別テーブルへスライドする設計 ― 使えるが、強い薬

ユーザーが指摘した通り、状態ごとにテーブルを用意し、状態遷移時に行を移動させる設計も成立する。これは「1つの表に状態を持つ」の反対側の極であり、状態ごとにデータ構造が大きく違う場合には有効。ただし、同一性・参照整合・履歴・検索に強い副作用がある。

34承認ワークフローの難所パターン ― 承認結果だけでなく、承認作業をモデル化する

承認は approved_by と approved_at だけでは足りないことが多い。実務には、差し戻し、再申請、代理承認、合議、条件付き承認、期限切れ、自動承認、並列承認、承認者変更、承認ルート改定がある。これらを「申請の状態」だけで持つと、承認作業の事実が消える。

承認モデルの層

層表の例役割注意
対象業務change_request, membership_application承認される対象そのもの承認のためだけの列で肥大化させない
承認インスタンスapproval_instance1回の承認プロセス全体対象業務IDを汎用参照にするか、業務別に表を分けるか判断
承認ステップapproval_step部長承認、品質承認、経理承認などの段階順序、並列、n-of-m、スキップ条件を持つ
承認タスクapproval_task誰が処理すべき作業か代理・再割当・期限・通知と相性がよい
承認アクションapproval_action承認・却下・差戻・コメントなどの事実追記専用にする。修正は取消/訂正アクション
承認定義approval_route_def, approval_ruleルートの版管理実行時には定義をスナップショット化する

なぜ承認定義の版が必要か

承認ルートは運用中に変わる。今月から品質部門の承認が追加されたとしても、先月開始した申請を新ルートで扱うとは限らない。承認インスタンス開始時点で「どのルート定義のどの版を使ったか」を固定しないと、過去の申請を再表示した時に、存在しなかった承認ステップが突然現れる。

機序: ワークフロー定義とワークフロー実行は別物
定義表は「これから作る承認プロセスの雛形」、インスタンス表は「実際に開始された承認プロセス」。定義変更で過去のインスタンスが変わると監査不能になる。実行時には、定義のIDだけでなく必要な承認者・条件・順序をスナップショットとして持つか、定義版を不変化する。

並列承認とn-of-m承認

実務でありがちな「3人中2人が承認すればよい」は、単純なステータスでは表現しづらい。ステップに required_approval_count を持ち、タスク単位のアクションを集計してステップ完了を判定する。

  step_no NUMBER NOT NULL,

  required_approval_count NUMBER,

35権限・アカウント・会員資格の分離 ― userテーブルに全部持たせない

「ユーザー」という言葉は雑に使われやすい。ログイン主体、人物、会員資格、従業員、顧客、権限ロール、認証プロバイダのアカウントは別概念である。特に退会・停止・組織異動・外部ID連携があるシステムでは、この分離が後の保守性を決める。

代表的な分離モデル

表意味主な列変化の仕方
party個人・法人・組織など、契約や所有の主体party_type, canonical_name名寄せ・分割・承継がある
person_profile個人属性name, birth_date, contact_id訂正・匿名化がある

external_identity外部IdPや外部システム上の識別子provider, external_subject, valid_from連携解除・再連携・ID変更がある

なぜ分けるのか

ログイン停止は、必ずしも会員資格の終了ではない。未払いで一時停止しても会員契約は残ることがある。退会しても、請求履歴や問い合わせ履歴の主体として人物または匿名化された主体は残ることがある。SSO連携を解除しても、内部会員資格は残る。これらをすべて users.status に押し込むと、状態数が爆発する。

状態の直積爆発

軸値の例混ぜた場合の問題
認証状態有効、ロック、パスワード期限切れ、MFA未設定、外部IdP停止会員状態と混ぜると「退会済みだがログインロック中」など意味不明な複合状態が出る
会員資格申請中、会員中、退会予定、退会済み、再入会済み認証・請求・権限の都合で状態値が増え続ける
権限一般、管理者、承認者、代理承認者、閲覧のみロール変更が会員状態変更として記録され、監査が崩れる
規約同意未同意、同意済み、再同意待ち同意待ちをログイン不可にするか会員停止にするかが曖昧
制裁・制限一時停止、永久BAN、本人確認待ち、要審査退会とは別の再登録防止・審査履歴を失う
機序: 状態軸を分けないと組合せが増殖する
5つの軸がそれぞれ4状態を持つと、理論上は4^5=1024通りの状態がある。実装者はその全てを user_status の列挙値で表すことはできない。軸ごとに表・列・履歴を分け、最終的な利用可否はビューや判定関数で導出する方が保守しやすい。

利用可否は状態の合成ビューにする

CREATE VIEW v_account_effective_access AS
SELECT a.account_id,
       m.membership_id,
       CASE
         WHEN a.auth_status != 'ENABLED' THEN 'DENY_AUTH'
         WHEN CURRENT_DATE NOT BETWEEN m.valid_from AND m.valid_to THEN 'DENY_MEMBERSHIP'
         WHEN EXISTS (SELECT 1 FROM account_restriction r
                     WHERE r.account_id = a.account_id
                       AND r.valid_from <= SYSTIMESTAMP
                       AND r.valid_to   >  SYSTIMESTAMP) THEN 'DENY_RESTRICTION'
         ELSE 'ALLOW'
       END AS access_decision
FROM account a
LEFT JOIN membership m ON m.party_id = a.party_id;

このように、元データは軸ごとに分け、画面やAPIが欲しい「使える/使えない」はビューで合成する。ビューは制約の代わりではないが、読み取り側の複雑さを局所化できる。

おすすめ
user という表名を使う場合でも、何のユーザーかを明確にする。認証なら account、会員資格なら membership、人物なら person、外部主体なら external_identity と分ける方が、退会・停止・再入会・権限移譲に強い。

36二時間軸・訂正・遡及反映 ― 「いつ起きたか」と「いつ知ったか」を分ける

履歴設計で最も見落とされるのは、時間が1種類ではないこと。業務上の有効日、実際の発生日、システム登録日、承認日、外部システム連携日、訂正日がずれる。これを1つの created_at や updated_at で代用すると、遡及訂正や過去時点再現ができない。

時間列の種類

時間意味例設計上の注意
発生時刻出来事が現実に起きた時刻入金日時、出庫日時、事故発生日イベントに必須。後から登録されることがある
登録時刻DBに記録された時刻データ取込日時、申請入力日時監査・遅延検知に必要
有効期間業務上その値が有効な期間価格の適用期間、所属の在籍期間半開区間がおすすめ
承認時刻業務的に有効化された時刻価格改定が承認された日時発生・登録・有効開始と一致しない
システム期間DB上でその行がその内容だった期間履歴表の recorded_from/to訂正前の誤った記録を再現できる
連携時刻外部に送信・受信した時刻ERP送信日時、会計IF取込日時再送・重複・失敗とセットで管理

39排他的FK・多態参照の追加妙技 ― 「どれか1つを参照する」をどう守るか

排他的外部キーは、RDBの弱点が出やすい。1つの行が、注文・申請・部品改訂・退会ケースのどれか1つだけを参照するような場合、単純なFKでは表現しづらい。実務では複数の妥協案を使い分ける。

方式比較

型 + IDtarget_type, target_id弱い高いログ・通知・検索など参照が緩いもの
スーパータイプ表approvable_object を作り、各対象が1:1でぶら下がる強い中承認対象・添付対象など共通扱いが多い

型 + ID方式を使ってよい条件

target_type + target_id はFKが効かないため、原則として正本データには避ける。ただし、通知ログ、操作ログ、検索インデックス、監査補助、外部リンクのように「参照先が消えてもログとして意味が残る」場合は許容されることがある。

警告

排他的FKの調定クエリ例

-- 型+ID方式を使った場合の調定例
SELECT l.*
FROM generic_link l
WHERE (l.target_type = 'ORDER' AND NOT EXISTS (SELECT 1 FROM orders o WHERE o.order_id = l.target_id))
   OR (l.target_type = 'APPLICATION' AND NOT EXISTS (SELECT 1 FROM membership_application a WHERE a.application_id = l.target_id))
   OR (l.target_type = 'PART_REVISION' AND NOT EXISTS (SELECT 1 FROM part_revision r WHERE r.part_revision_id = l.target_id));
おすすめ
対象が3〜5種類程度で固定なら nullable FK + CHECK。共通操作が多いならスーパータイプ。対象が多く、参照がログ的・検索的なら型+IDも可。ただし正本の業務整合性が必要な関連で型+IDを使う場合は、かなり慎重に扱う。

40一意性がないデータの受け皿設計 ― 無理にユニーク制約を作らない

現実には、ユニークキーが存在しないデータがある。氏名・住所・電話番号・メール・品名・外部ファイル行番号・図番の一部は、単独では一意にならない。無理に「それっぽい複合キー」を作ると、重複・誤統合・取込不能の原因になる。

台帳 + 残高キャッシュの例

CREATE TABLE stock_ledger (
  stock_ledger_id NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  item_id NUMBER NOT NULL,
  location_id NUMBER NOT NULL,
  event_type VARCHAR2(30) NOT NULL CHECK (event_type IN ('RECEIPT','ISSUE','ADJUSTMENT','RESERVE','RELEASE','SHIP')),
  quantity_delta NUMBER NOT NULL,
  occurred_at TIMESTAMP NOT NULL,
  source_doc_type VARCHAR2(40),
  source_doc_id NUMBER,
  reversal_of_ledger_id NUMBER REFERENCES stock_ledger(stock_ledger_id)
);

CREATE TABLE stock_balance (
  item_id NUMBER NOT NULL,
  location_id NUMBER NOT NULL,
  onhand_qty NUMBER NOT NULL,
  reserved_qty NUMBER NOT NULL,
  updated_at TIMESTAMP NOT NULL,
  PRIMARY KEY (item_id, location_id)
);

台帳は追記専用で、残高は同一トランザクションで更新する。残高は台帳から再計算できる冗長データであるため、定期的に調定する。

調定クエリ例

SELECT b.item_id, b.location_id,
       b.onhand_qty AS cached_onhand,
       SUM(CASE WHEN l.event_type IN ('RECEIPT','ADJUSTMENT') THEN l.quantity_delta
                WHEN l.event_type IN ('ISSUE','SHIP') THEN l.quantity_delta
                ELSE 0 END) AS ledger_onhand
FROM stock_balance b
JOIN stock_ledger l
  ON l.item_id = b.item_id
 AND l.location_id = b.location_id
GROUP BY b.item_id, b.location_id, b.onhand_qty
HAVING b.onhand_qty != SUM(CASE WHEN l.event_type IN ('RECEIPT','ADJUSTMENT','ISSUE','SHIP') THEN l.quantity_delta ELSE 0 END);

落とし穴説明対策

期限切れを考えないカート放置で永遠に在庫が減るreserved_until と期限切れ解放ジョブ
残高更新をアプリで読んで計算ロストアップデートの温床DB内の差分UPDATE、行ロック、楽観ロック
負数を許すか曖昧バックオーダー可否が不明品目・倉庫・業務ごとのポリシーを持つ

おすすめ

データ削除は単なる DELETE ではない。業務上の削除、画面からの非表示、ログイン不可、個人情報の匿名化、法定保持期限切れ、監査証跡の保持は別物である。削除設計を後回しにすると、データ消失か個人情報過保持のどちらかに倒れる。

CREATE TABLE deleted_object_tombstone (

tombstoneは、外部同期で「削除されたこと」を伝える、ID再利用を防ぐ、再取込時に復活させない、といった用途に使える。実体データは消しても、削除済みである事実は残せる。

監査や請求のために取引履歴は残したいが、退会者の氏名・メール・住所は消したい場合がある。このとき、取引履歴のFK先を物理削除すると壊れる。代わりに、人物IDは残し、個人特定列を匿名化する、または匿名化済みpartyへ付け替える。

方法利点欠点

匿名化テーブルへ分離PIIの所在を限定できるJOINと権限設計が必要
暗号化 + 鍵破棄履歴行を残しつつ復号不能にできる鍵管理・法的解釈・運用が難しい
集計値だけ残す個人情報リスクが低い個別問い合わせに答えられない

保持期限をテーブル設計に入れる

CREATE TABLE retention_policy (
  retention_policy_id NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  object_type VARCHAR2(60) NOT NULL,
  retention_basis VARCHAR2(60) NOT NULL,
  retention_months NUMBER NOT NULL,
  action_after_retention VARCHAR2(30) NOT NULL CHECK (action_after_retention IN ('ARCHIVE','ANONYMIZE','DELETE','REVIEW'))
);

保持期限は運用手順だけでなく、データに紐づくポリシーとして持つと管理しやすい。退会日から何年、最終取引日から何年、法定保存期間終了まで、など起算点が違うため、対象ごとに設計する。

警告
論理削除列を足すだけでは削除設計にならない。どの画面から除外するか、どの帳票には残すか、外部連携には削除を送るか、バックアップやアーカイブからも消す必要があるか、保持期限後に何をするかまで決める。

イベントソーシングは強力だが重い。すべてのエンティティをイベントソースにする必要はない。実務では、純粋なCRUD、監査ログ併用、ドメインイベント、Outbox、台帳、完全イベントソーシングを使い分ける。

採用レベル

イベントは「状態が変わった」ではなく、できるだけ業務意図を表す。StatusChanged より MembershipApplicationApproved、WithdrawalRequested、PartRevisionReleased の方が後から意味を解釈しやすい。ただし粒度を細かくしすぎるとイベント種類が爆発する。

46外部連携・取込・冪等性 ― 連携データは信用せず、再実行できるように受ける

外部連携では、同じデータが複数回来る、順序が前後する、途中で失敗する、外部側のIDが変わる、取消が後から来る。DB設計で冪等性と取込履歴を持たないと、再実行のたびに重複・欠落・二重処理が起きる。

冪等性キー

冪等性とは、同じ処理を何度実行しても結果が同じになる性質である。外部APIやファイル取込では、source_system、external_message_id、business_event_id、payload_hash などを使って重複処理を防ぐ。

CREATE TABLE inbound_message (
  inbound_message_id NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  source_system_code VARCHAR2(50) NOT NULL,
  external_message_id VARCHAR2(200) NOT NULL,
  payload_hash VARCHAR2(128) NOT NULL,
  received_at TIMESTAMP NOT NULL,
  process_status VARCHAR2(30) NOT NULL,
  processed_at TIMESTAMP,
  CONSTRAINT uk_inbound_message UNIQUE (source_system_code, external_message_id)

外部から「変更」「削除」「作成」が順番通りに来るとは限らない。削除が先に来て、後から作成が届くこともある。対策は、外部イベントの発生時刻・シーケンス番号・バージョンを持ち、古いイベントを無視または保留できるようにすること。

対策内容注意
外部バージョン外部エンティティの版番号を比較し、新しいものだけ反映外部が単調増加を保証する必要
発生時刻新旧を時刻で判断時計ずれ・遅延登録に注意
保留キュー親が来ていない子イベントを保留期限切れ・手動修復が必要
再同期差分ではなくスナップショットで整合を取り直す全件負荷と整合窓
調定外部と内部の件数・ハッシュ・重要項目を突合日次・週次など運用設計が必要

Outboxパターン

DB更新とメッセージ送信を別トランザクションにすると、「DBは更新されたがメッセージ送信に失敗」「メッセージだけ送られたがDBがロールバック」という事故が起きる。Outboxでは、DB更新と同じトランザクションで送信予定イベントを保存し、別プロセスが送信する。

CREATE TABLE outbox_event (
  outbox_event_id NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  aggregate_type VARCHAR2(60) NOT NULL,
  aggregate_id NUMBER NOT NULL,
  event_type VARCHAR2(100) NOT NULL,
  payload CLOB NOT NULL,
  created_at TIMESTAMP NOT NULL,
  publish_status VARCHAR2(30) NOT NULL CHECK (publish_status IN ('PENDING','PUBLISHED','FAILED','DEAD_LETTER')),
  published_at TIMESTAMP,
  retry_count NUMBER DEFAULT 0 NOT NULL
);
おすすめ
外部連携は「成功する前提」で設計しない。メッセージID、取込バッチ、ステージング、外部ID履歴、冪等性キー、再実行可能な処理、調定クエリを最初から置く。連携の失敗を業務データの不整合にしない。

tenant_id漏れ対策

  • テナント配下の全表で、PKを (tenant_id, local_id) にするか、少なくとも一意制約に tenant_id を含める。
  • FKにも tenant_id を含め、他テナント行を参照できないようにする。
  • 横断管理者用ビューと通常利用者用ビューを分ける。
  • テナント境界のないグローバル表を明示的に分類する。
  • テストデータで「同じlocal_idが別テナントに存在する」ケースを必ず作る。
警告
マルチテナントで最も危険なのは、JOIN条件の tenant_id 漏れで他社データが見えること。これは性能問題ではなく重大なセキュリティ事故である。設計段階で複合FK、ビュー、RLS/VPD、テスト観点を入れる。

48A申請・ドラフト・提出スナップショット ― 「今のマスタ」と「申請時の内容」を混ぜない

入会申請、部品改訂申請、価格改定申請、退会申請のような申請系データでは、申請者が入力した時点の内容と、現在のマスタ値がずれる。申請が承認された時に、どの内容を承認したのかを再現できない設計は危険である。

申請系でよくある状態

状態意味編集可否DB設計上の扱い
ドラフト本人または担当者が編集中可上書き可。ただし下書き保存履歴が必要なら別途版管理
提出済み承認プロセスへ渡した原則不可提出時スナップショットを固定
差戻中承認者から修正依頼可再提出で申請版を増やす
承認済み業務的に採用された不可承認対象内容を不変として保存
却下採用されなかった不可または再申請へコピー却下理由と却下時点内容を残す

方式形強い点弱い点向くケース

OracleのIndex Organized Tableは、主キー順で行を格納するため、主キーまたは主キーprefixでの範囲アクセスが多い表に向くことがある。たとえば、(aggregate_id, sequence_no) でイベントを読むイベントストリーム、(tenant_id, code) の小さな参照表など。ただし、非キー列が大きい表、二次索引が多い表、更新が多い表では慎重に検討する。

おすすめ
最初から詳細なストレージ設計を固める必要はないが、履歴・イベント・ログ・取込rawのように増え続ける表では、日付列、テナント列、取込バッチIDなど、将来のパーティションキー候補を設計に入れておく。後から列を足すより安全である。

# ADR: 会員状態の保持方式

## Status
Accepted / 2026-06-02

## Context
会員には申請中、会員中、一時停止、退会済み、再入会がある。
退会時には請求・個人情報削除・外部連携停止が絡む。
状態ごとに必要な列が異なるが、全ユーザー横断検索も必要。

## Decision
単一の users 表に全状態を詰め込まず、party / account / membership / withdrawal_case を分ける。
membership には現在状態を持ち、membership_status_event に状態遷移を追記する。
退会手続きは withdrawal_case として独立させる。

## Alternatives considered
1. users.status のみ
   - 読みは簡単だが、退会ケース・再入会・監査が弱い。
2. 状態別テーブルへ完全スライド
   - 状態固有制約は強いが、横断検索とFKが難しい。
3. 完全イベントソーシング
   - 監査は強いが、現時点のチーム・要件では過剰。

## Consequences
- JOINは増える。
- 状態遷移時は現在値と履歴を同一トランザクションで更新する。
- 将来、承認ルートが複雑化したら approval_instance を追加する。

COMMENT ON TABLE membership IS
'会員資格を表す。人物(person/party)や認証アカウント(account)ではない。再入会時は原則として別membership行を作る。';

COMMENT ON COLUMN membership.valid_from IS
'会員資格の業務上の有効開始日。DB登録日時ではない。';

COMMENT ON COLUMN membership.valid_to IS

元の意図意図が消えた後の変更事故

複雑な設計ほど、DDLだけでなく「なぜそうしたか」を短いADRで残す。特に、非正規化、JSON、型+ID、状態別テーブル、履歴表、投影表、外部ID履歴、論理削除は、意図が失われると誤用されやすい。

49まとめと参考資料

テーブル設計の上級者が見ているのは、列の型だけではない。意味、同一性、時間、制約、履歴、削除、読取、移行、運用を同時に見ている。

要点

  • リソース、イベント、状態、履歴、射影を分ける。混ぜると過去と理由が消える。
  • 変更される可能性のある外部IDや業務番号をPKにしない。内部代理キーと対応表で守る。
  • 状態管理は現在値だけでなく、状態遷移履歴・承認イベント・承認タスクを分ける。
  • ユニークキーがないデータは、rawを保持し、名寄せ候補と確信度を管理する。取り込み時に捨てない。
  • 排他的外部キーや多態参照は、柔軟性と参照整合性のトレードオフを明示する。
  • 型別データは、共通表 + 型別1:1表 + JSON自由属性の三層で考える。
  • 履歴・時系列は、発生時刻、登録時刻、有効期間を分ける。半開区間を基本にする。
  • 非正規化、MV、CQRS、イベントソーシングは、書込モデルの真実を守った上で読取を最適化するために使う。
  • 削除、訂正、取消、終了、匿名化を混同しない。物理削除は再生成可能なデータに限定する。
  • スキーマは必ず変わる。expand/contract、NOVALIDATE、EBR、オンライン再定義などを前提に、互換期間を設計する。
