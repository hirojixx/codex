## 1. 調査質問

- 「`new Child(this)`」のように、集約クラス（親）参照を内部クラス（子）に渡す設計は、どの局面で有効か。
- よくある実装例と利用箇所を、判断に使える粒度で整理できるか。
- 採用/不採用の選定を、実例ベースで説明できるか。

## 2. 結論（先に短く）

- 親参照を渡す設計は、**API 集約を維持したまま内部責務を分割する**ための実践的な手段。
- ただし「何でも親を呼べる」状態にすると失敗する。成功条件は次の 3 つ。
  1. 子へ渡すのは「親そのもの」ではなく、可能なら最小 interface。
  2. 親子のライフサイクル（open/flush/close/retry）を状態遷移として定義。
  3. 再入（child -> parent -> child）と循環依存のガードを設ける。

## 3. パターンの意味（再整理）

### 3.1 何を解決するか

- **Facade + Delegation**: 外部 API は親に残し、実処理は子に委譲。
- **Context 共有**: 子は親経由で設定・認証・ID・ロガーへアクセス。
- **God Class 抑制**: キュー制御/リトライ/整列送信などを子へ隔離。

### 3.2 どこで破綻するか

- 子が親の多くの機能を使い始める（責務リーク）。
- 初期化順依存（子生成時に親が未完成）。
- close/flush の責任境界が曖昧（データロス/ハング）。
- 再入経路で無限ループ/多重実行。

## 5. 他手法との比較（調査10件）

| # | 比較対象 | 向いている状況 | このパターンより優れる点 | このパターンが優れる点 |
|---|---|---|---|---|
| 1 | 純粋DI（親参照なし） | 依存方向を単方向に固定したい | 循環依存を作りにくい | API入口の集約は弱くなりやすい |
| 2 | Event Bus | 疎結合で非同期連携したい | 変更波及を抑えやすい | 制御フロー追跡は難しくなる |
| 3 | Mediator | 多数コンポーネントの調停が必要 | 相互参照を抑制できる | 中央調停者が肥大化しやすい |
| 4 | Domain Service 分離 | ドメイン規則を独立させたい | テストしやすい純粋ロジック化 | API境界と実装が分かれすぎることがある |
| 5 | Functional Core / Imperative Shell | 副作用を外殻に閉じたい | 予測可能性が高い | 実装ボイラープレートが増える |
| 6 | Strategy 注入 | アルゴリズム差し替えが多い | 実行戦略を容易に交換 | 親子で共有文脈を持ちづらい |
| 7 | Repository + UoW | 永続化境界を厳密化したい | 取引整合性を保ちやすい | I/Oオーケストレーションは別途必要 |
| 8 | Actor Model | 高並行・分散で状態隔離したい | 競合回避に強い | 同期的APIの単純性は下がる |
| 9 | Hexagonal Architecture | 外部I/O差し替えを重視 | ポート/アダプタで保守しやすい | 小規模では構成過多になりがち |
| 10 | Plugin/Hook 方式 | 拡張点を外部公開したい | 拡張性が高い | 実行順・互換性管理が難しい |

**使い分けの実務目安**
- 親APIを中心に据えて「同期的で分かりやすい利用体験」を優先するなら、本パターン（親参照注入）を第一候補にする。
- 依存グラフの健全性・将来拡張・疎結合を最優先するなら、DI / Event Bus / Hexagonal を先に検討する。
- 高並行処理が主課題なら Actor、アルゴリズム差し替えが主課題なら Strategy を優先する。

## 6. 選定手順（実務向け 7 ステップ）

1. **外部API集約の必要性**: 利用者に見せる入口を1つにする価値があるか。
2. **内部責務の独立性**: 子クラスごとにテスト可能な責務があるか。
3. **依存の最小化設計**: `ParentLike` の最小 interface を定義できるか。
4. **状態遷移設計**: `init -> running -> draining -> closed` を文書化できるか。
5. **再入ガード**: 親子呼び出しループの遮断策を実装できるか。
6. **障害時方針**: 子が例外時に親へ通知し、親が最終判断する形にできるか。
7. **テストコスト試算**: 親モックが巨大化しないか（失敗の早期検知）。

## 7. 選定の実例（2ケース）

### ケースA: 採用した方がよい

- 条件:
  - 公開APIは `Topic.publish()` だけを見せたい。
  - 内部で batching / ordering / flow control を分離したい。
  - close時に未送信データを親が一括 flush したい。
- 判断:
  - **採用**。ただし `Publisher` が触れる親機能は `TopicPublishContext` に限定。

### ケースB: 採用しない方がよい

- 条件:
  - 子が親の更新系メソッドを多数呼ぶ（設定更新、接続更新、監視更新…）。
  - 親生成時にネットワーク初期化が多段。
- 判断:
  - **不採用**。DI + domain service + event bus に分離した方が保守性が高い。

## 8. 実装テンプレート（最小インタフェース版）

```ts
interface TopicPublishContext {
  getRequestContext(): { topicName: string };
  onPublisherError(err: Error): void;
}

class Topic implements TopicPublishContext {
  private readonly publisher: Publisher;

  constructor(/* options */) {
    // 必須状態を先に初期化
    this.publisher = new Publisher(this);
  }

  getRequestContext() {
    return { topicName: 'projects/p/topics/t' };
  }

  onPublisherError(err: Error) {
    // 親で最終判断（retry/abort/metrics）
  }

  publish() {
    return this.publisher.publish();
  }
}

class Publisher {
  constructor(private readonly ctx: TopicPublishContext) {}

  publish() {
    const req = this.ctx.getRequestContext();
    // 実送信
    // エラー時: this.ctx.onPublisherError(err)
    return req;
  }
}
```
