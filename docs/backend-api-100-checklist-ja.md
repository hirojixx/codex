# バックエンドAPI設計・実装の障害予防チェックリスト100（Node.js含む）

> 目的: 一般的なバックエンドAPIで障害が起こりやすい箇所を、設計・実装・運用・デバッグまで横断して整理する。

> 再構成版（運用しやすい要約）は `docs/backend-api-checklist-playbook-ja.md` を参照。
## 1. ドメイン設計（DDD含む）
1. **ユビキタス言語**が仕様書・コード・API項目名で一致しているか。
2. **境界づけられたコンテキスト**を分割し、責務の越境を避けているか。
3. **集約(aggregate)の整合性境界**を明示し、トランザクション境界と整合しているか。
4. **エンティティ識別子の生成戦略**（UUID/ULID/DB採番）を決めているか。
5. **値オブジェクトの不変条件**を型・バリデーションで強制しているか。
6. **ドメインサービス**とアプリケーションサービスの責務を混同していないか。
7. **ドメインイベント**の発火条件・再送方針・順序要件を定義しているか。
8. **読み取りモデル(CQRS)**を導入する場合の整合性遅延を許容できるか。
9. **マルチテナント境界**（データ分離/権限/課金）をモデル上で表現しているか。
10. **時刻とタイムゾーン**（UTC保存、表示時変換）のルールを統一しているか。

## 2. API契約設計
11. リソース設計がREST/GraphQL/gRPC方針に沿って一貫しているか。
12. URL命名規約（複数形、階層深さ、ID位置）を固定しているか。
13. HTTPメソッドの意味（GET安全性、PUT冪等性など）を守っているか。
14. **ステータスコード設計**（4xx/5xx、409/422の使い分け）が明確か。
15. エラーレスポンス形式（code/message/details/traceId）を標準化しているか。
16. ページネーション方式（offset/cursor）と一貫したソートキーを定義しているか。
17. フィルタ・検索パラメータの型/演算子/上限を決めているか。
18. APIバージョニング戦略（URI/ヘッダ/互換期間）を定義しているか。
19. 後方互換性ポリシー（追加は可、削除は猶予など）を運用ルール化しているか。
20. OpenAPI/AsyncAPI/Protoで契約を機械可読化しているか。

## 3. 入出力バリデーション
21. 入力スキーマ検証（必須、型、長さ、範囲）を実装しているか。
22. 文字コード・正規化（NFC/NFKC）を考慮しているか。
23. 日付/時刻のパース失敗時の扱いが統一されているか。
24. 予期しないフィールドを拒否（allowlist）しているか。
25. JSONスキーマやzod/io-ts等で型定義をランタイム検証しているか。
26. ファイルアップロード時のMIME・拡張子・サイズ検証をしているか。
27. 出力のマスキング（PII、秘密情報）を適用しているか。
28. 浮動小数誤差を避けるため金額を整数最小単位で扱っているか。
29. null/undefined/空文字の意味を仕様で分離しているか。
30. 多言語入力（絵文字、サロゲートペア）の保存・検索テストをしているか。

## 4. 認証・認可・セッション

## 障害が起こりやすい典型ポイント（優先監視）
- 競合更新（同時実行）による二重処理・残高不整合。
- 外部API遅延でスレッド/接続枯渇し連鎖障害化。
- キャッシュ不整合・キー設計ミスによる古いデータ返却。
- マイグレーション手順不備によるデプロイ後エラー。
- Node.jsイベントループ詰まりで全体遅延増大。
- メモリリークでOOM再起動ループ。
- 認可漏れ(BOLA)による情報漏えい。

## 障害対応の最小フロー（実務向け）
1. **検知**: アラート受信、影響範囲（ユーザー/API）を即時把握。
2. **緩和**: Feature Flag停止、レート制限強化、ロールバック。
3. **切り分け**: 直前変更、依存先、メトリクス差分を確認。
4. **復旧**: 安全策（リードオンリー化、キュー滞留解消）で復旧優先。
5. **恒久対策**: 根本原因、再発防止、テスト追加、Runbook更新。

## すぐ使えるデバッグ観点（Node.js）
- レイテンシ急増時: event loop lag、GC時間、外部I/O待ち。
- メモリ増加時: heap snapshot比較、保持参照、キャッシュ肥大。
- スループット低下時: DB待ち、ロック競合、プール枯渇。
- 5xx増加時: 依存先タイムアウト、リトライ暴走、入力異常。

## Web調査ベース追加108件（151-258）

> 主要参照: OWASP API Security Top 10 (2023), RFC 9110/9111/9112/9333/9457, RFC 8725/8705/9449, Kubernetes Probe docs, OpenTelemetry SemConv, Node.js Diagnostic Report, NIST SP 800-61r3。

### 16. HTTP仕様・互換性（RFC準拠）
151. `GET/HEAD/OPTIONS`を**安全メソッド**として実装し、副作用を持たせていないか。
152. `PUT`/`DELETE`の冪等性を壊す実装（内部で二重課金等）を排除しているか。
153. `POST`に冪等キーを要求するエンドポイントを明示しているか。
154. `ETag`/`If-Match`で更新競合（Lost Update）を防いでいるか。
155. `Cache-Control`/`Vary`を誤用して認可情報を共有キャッシュしていないか。
156. `429 Too Many Requests`と`Retry-After`で過負荷時の挙動を明示しているか。
157. `503 Service Unavailable`時の再試行指針をクライアントへ返しているか。
158. `Content-Type`と実データ形式の不一致を拒否しているか。
159. 大きすぎる入力に`413 Payload Too Large`を返し上限を文書化しているか。
160. エラー形式を`application/problem+json`（RFC 9457）で標準化しているか。
161. 条件付き取得（`If-None-Match`）を活用し不要転送を抑制しているか。
162. 圧縮爆弾対策として解凍後サイズ上限をチェックしているか。

### 17. APIセキュリティ（OWASP API Top 10 2023対応）
163. API1(BOLA)対策としてID参照ごとの認可判定を実装しているか。
164. API2(Broken Authentication)対策としてトークン管理を厳密化しているか。
165. API3(BOPLA)対策として返却項目の露出を最小化しているか。
166. API4(Unrestricted Resource Consumption)対策として計算量制限を設けているか。
167. API5(BFLA)対策として機能レベル認可を全ロールで検証しているか。
168. API6(Unrestricted Access to Sensitive Business Flows)に対し業務フロー保護をしているか。
169. API7(SSRF)対策として送信先制御・DNS rebinding対策をしているか。
170. API8(Security Misconfiguration)としてデフォルト設定放置を排除しているか。
171. API9(Improper Inventory Management)として公開API棚卸しを維持しているか。
172. API10(Unsafe Consumption of APIs)として依存APIの検証・契約確認をしているか。
173. GraphQL利用時に深さ制限/複雑度制限を実装しているか。
174. APIキー漏えい検知（ハニートークン等）を導入しているか。

### 18. 認証基盤・トークン保護（OAuth/JWT）
175. Bearerトークン利用時、TLS必須・平文保存禁止を徹底しているか。
176. JWTで`alg`混乱攻撃を避けるため受理アルゴリズムを固定しているか。
177. JWTの`aud`/`iss`/`exp`/`nbf`/`iat`を厳格検証しているか。
178. 鍵ID(`kid`)運用とJWKSローテーション失敗時のフォールバックを設計しているか。
179. リフレッシュトークンの再利用検知（rotation+reuse detection）を実装しているか。
180. Token binding手法（mTLS: RFC 8705 / DPoP: RFC 9449）を要件に応じ検討しているか。
181. スコープ最小化（least privilege）をクライアント単位で定義しているか。
182. 機械間認証（client credentials）に証明書ベース認証を検討しているか。
183. API Gatewayとアプリ層で認証責務が重複・乖離していないか。
184. 認証失敗時の応答差異からアカウント存在有無が推測できないか。
185. セッション失効とトークン失効の整合（ログアウト一貫性）を確認しているか。
186. 時刻ずれ（clock skew）許容幅を明示し検証しているか。

### 19. レート制御・トラフィック管理
187. 利用者ごとに公平なレート制御キー（user/app/ip）を定義しているか。
188. 重要APIに優先キューを設け、非重要APIで飢餓を起こさないか。
189. IETF RateLimitヘッダ（RFC 9333）採用可否を決めているか。
190. Retry-Afterを秒とHTTP-dateの両形式で解釈可能にしているか。
191. クライアントSDKへバックオフ+jitter実装を提供しているか。
192. サーバ側リトライ禁止エラー（4xx）を明確にしているか。
193. バースト許容量と定常レートを分離して設計しているか。
194. テナント単位での過負荷隔離（noisy neighbor対策）をしているか。
195. キャッシュスタンピード時にsingle-flight等で抑制しているか。

198. 外部依存先のQPS制限を内製API制限に反映しているか。

### 20. 可観測性・テレメトリ標準化
199. OpenTelemetryでHTTP span属性をSemConvに沿って統一しているか。
200. 高カーディナリティ属性（userId等）の無制限出力を防いでいるか。
201. トレースのサンプリング戦略を障害時に切替可能にしているか。
202. ログ/メトリクス/トレースで同一`trace_id`連携を保証しているか。
203. ヘッダ収集はallowlist方式で機密漏えいを防いでいるか。
204. 依存先再試行回数をメトリクス化しリトライ暴走を検知しているか。
205. SLO違反予兆（エラーバジェット消費率）を可視化しているか。
206. メトリクス粒度（1m/5m）と保持期間を用途別に設計しているか。
207. オンコール向けダッシュボードを「影響度順」に構成しているか。
208. 障害時に一時的デバッグログを安全に増やす仕組みがあるか。
209. 設定変更（feature flag含む）を監査イベントとして記録しているか。
210. 監視停止そのものを検知するハートビート監視を入れているか。

### 21. Node.js 実践追加（運用障害寄り）
211. `stream.pipeline()`を利用しエラー伝播漏れを防いでいるか。
212. Stream破棄時の`destroy()`連鎖とFDリークを検証しているか。
213. `AbortSignal`連携でクライアント切断時に処理停止しているか。
214. `server.headersTimeout`/`requestTimeout`を攻撃耐性込みで設定しているか。
215. `keepAliveTimeout`とLB設定の不整合で切断多発していないか。
216. 診断レポート（`--report-*`）を本番障害時に採取できるか。
217. ヒープスナップショット採取手順をRunbook化しているか。
218. `UV_THREADPOOL_SIZE`調整がI/O負荷特性に合っているか。
219. 暗号/圧縮/画像処理等CPU負荷処理をプロセス分離しているか。
220. NodeのLTSアップデートを四半期単位で検証しているか。
221. unhandled rejectionの終了ポリシーを環境別に統一しているか。
222. SIGTERM受信時にgraceful shutdown（接続停止→完了待ち）をしているか。

### 22. コンテナ/Kubernetes運用
223. startup/readiness/livenessの役割を分離し誤再起動を防いでいるか。
224. readiness失敗時に依存障害を正しく反映しトラフィック遮断できるか。
225. liveness過敏設定で高負荷時の連鎖再起動を起こしていないか。
226. Pod終了時のdrain猶予（preStop/terminationGracePeriod）を確保しているか。
227. リソースrequests/limitsが実負荷と乖離していないか。
228. HPA指標がCPU偏重で、キュー遅延を見落としていないか。
229. PodDisruptionBudgetで同時退避数を制限しているか。
230. ノード障害時の再配置でゾーン偏りを起こさない設定か。
231. ConfigMap/Secret更新がローリング反映される運用か。
232. サービスメッシュ導入時にタイムアウト/リトライ二重化していないか。
233. クラッシュループ時に自動ロールバック条件を設けているか。
234. メトリクス・ログ収集エージェント障害を別監視しているか。

### 23. 事業継続・インシデントマネジメント
235. NIST SP 800-61r3に沿って準備・検知・対応・学習を循環できるか。
236. 重大度定義（SEV1〜）とエスカレーション時間を定めているか。
237. 影響評価で「売上・法令・信用」を同時に判定できるか。
238. 法務・広報・CS連携のインシデント連絡網を整備しているか。
239. フォレンジック観点で証拠改変を防ぐ取得手順があるか。
240. 顧客告知のテンプレート（初報/続報/終報）を事前作成しているか。
241. 復旧優先順位（安全性 vs 可用性）を意思決定基準化しているか。
242. 復旧後に未処理データの再処理順序を定義しているか。
243. 取引先/SaaS障害時の代替フローを契約SLAと合わせているか。
244. 監査対応で「いつ何をしたか」を時系列で即提出できるか。
245. ポストモーテムのアクションに期限・責任者を必須化しているか。
246. 同種障害の再発率を四半期で計測し改善効果を追跡しているか。

### 24. データ/プライバシー/AI連携API
247. データ最小化原則に沿って収集項目を削減しているか。
248. PII項目の暗号化対象と鍵管理責任を明確化しているか。
249. 削除要求時にバックアップ・ログの扱いを規定しているか。
250. 学習用途への二次利用可否を同意管理と紐づけているか。
251. 監査ログの長期保存で検索性能劣化対策を行っているか。
252. 外部AI API利用時にプロンプト/応答の機密分類をしているか。
253. AI応答キャッシュ時にテナント越境漏えいを防いでいるか。
254. モデル更新時の挙動差分を回帰テストしているか。
255. 自動化判断（与信等）で説明可能性ログを保持しているか。
256. データ越境（リージョン外転送）を法令要件でチェックしているか。
257. 保持期限到達データの自動削除ジョブを監査しているか。
258. 匿名化/仮名化の再識別リスク評価を定期実施しているか。
