# 判断材料ドキュメント生成プロジェクト

## 目的

特定テーマについて、**「判断材料」**となる資料を作る。
できる限り多様な観点・著者・媒体から情報を集積し、**選択肢を提示する**ことを最優先する。
「飛ばし読み前提で情報量を最大化する」方針。まとめが必要な場合は都度個別指示で行う。

## 成果物

- **形式**: Single HTML(1ファイル完結)
- **構成**: 全観点を1つの巨大HTMLに集約。目次・章ジャンプ・観点別アンカーを付け、飛ばし読みを容易にする
- **画像**: すべて Base64 で埋め込み(外部依存なし)
- **保存先**: `./<テーマ名>/<テーマ名>.html`
- **付随物**: 調査ログ(収集ソース一覧・URL・著者/媒体・観点タグ)を同フォルダに残す

## ワークフロー(1サイクル)

### 1. 調査(Research)
- WebSearch を主軸に、**1テーマあたり最低100件**収集。**上限なし、必要なだけ**深掘りする
- **多様性の担保**(同じような著者に偏らせない):
  - 観点ごとにクエリを分割(下記「観点軸」参照)し、検索を分散
  - 立場の異なるクエリを意図的に投げる(賛成/反対、推進/慎重、商用/学術/個人、国内/海外 など)
  - 媒体タイプを散らす(学術・公的機関・業界紙・企業・個人ブログ・コミュニティ・一次資料)
  - 同一著者・同一ドメインの偏りを記録し、上位を独占させない
- 収集物は出典・著者・媒体タイプ・観点タグ付きで記録

### 2. 作成(Compose)
- **要約はしない。ただし、同一の情報量であれば言い回しの変更はOK。アイコンに変更等はOK**。観点別に情報を積み上げる
- **相反する見解を並列で残す**。「Aが有力」で閉じず、B/C/Dの選択肢と各々の根拠・反論・条件を併記

### 3. レビュー(Review)── 削るより「広げる」
- **明らかな誤り以外は残す**(情報量を最大化)
- **断定箇所を検出して指摘**: 有力候補が1つだけ確定的に書かれている箇所を探し、**他の選択肢を追加調査して広げる**
- 重複の機械的削除はせず、観点が異なるなら残す
- 多様性チェック: 著者/媒体が偏っていれば、不足する立場を追加調査
- このレビューは**数回繰り返す**(観点の網羅性が飽和するまで)

### 4. Single HTML 化(Build)
- 章立て・目次・アンカー・画像Base64埋込で1ファイル生成
- 出典等は`./<テーマ>/research/*.md`を確認するので記載は不要
- 情報を分かりやすくまとめること

## 観点軸(クエリ分散とレビューの網羅性チェックに使う雛形)

テーマに応じて取捨選択・追加する:

- 技術的観点 / 実装難易度
- コスト・経済性
- リスク・デメリット・失敗事例
- メリット・成功事例
- 代替案・競合選択肢の比較
- 法規制・コンプライアンス
- 時系列・トレンド・将来予測
- 賛成派の論拠 / 反対派の論拠
- 国内事情 / 海外事情
- 一次資料(統計・公的データ・原典)

## SubAgent オーケストレーション

調査フェーズは SubAgent で並列化する。**1サイクルあたり 4〜6 体**に集約する(近い観点をグループにまとめて1体に割り当てる)。

### 役割分担
- **親(オーケストレーター)**: 観点軸を 4〜6 グループに束ね、各グループへリサーチ SubAgent を1体ずつ割り当てる。目録を突き合わせ、追撃を指示し、最後に HTML へ統合する。
- **リサーチ SubAgent(general-purpose)**: 担当グループの検索戦略(クエリ群)・狙う媒体タイプ・立場の指定を受け、調査して**生情報をファイルに書き出す**。

### 最重要ルール ── 要約を返させない
SubAgent は終了時に短い要約しか親に返せない。これを情報の圧縮に使わない。各 SubAgent は:

- 生情報を `./<テーマ>/research/<観点グループ>.md` に書き出す(本文抜粋 + URL + 著者/媒体 + 媒体タイプ + 観点タグ + 取得日)。
- 親へ返すのは**目録のみ**: 件数、出典ドメイン一覧、著者一覧、カバーした論点、未カバーの論点。

### ノルマと多様性
- 件数は親がグループへ配分し、合計が最低100件に届くようにする(例: 5体なら各20〜25件以上)。上限なし。
- 同一ドメイン/著者がグループ内で過度に重複しないよう各 SubAgent に分散を指示。親は目録で事後確認し、偏り・観点の穴に追撃 SubAgent を割り当てる。

### 注意
- SubAgent は毎回コールドスタートで文脈を持たないため、割当プロンプトに「テーマ・担当観点・出力先パス・書式・要約禁止・多様性ルール」を明記する。

## 確定パラメータ(合意済み)

| 項目     | 設定                                                   |
|----------|--------------------------------------------------------|
| 出力構成 | 全観点を1つの巨大HTMLに集約                                |
| 調査量   | 最低100件前後、上限なし(必要なだけ)                          |
| 調査並列 | リサーチ SubAgent を 4〜6 体。生情報はファイル書き出し、親へは目録のみ    |
| レビュー強度 | 明らかな誤り以外は残す。断定的に1候補のみの箇所は選択肢を広げるよう指摘 |
| 画像     | Base64 埋め込み                                          |
| 要約     | 原則しない(必要時のみ個別指示)                              |
| 多様性   | 著者・媒体・立場を分散。偏りを記録し補正                      |

## 進め方

1. テーマを受け取る
2. 観点軸を 4〜10 グループに設定し、リサーチ SubAgent へ割り当て(各が research/ にファイル書き出し)
3. 親が目録で重複・偏り・観点の穴を確認し、不足に追撃 SubAgent
4. 親が全 research ファイルを読んで観点別に再構成して積み上げ
5. レビュー(広げる方向)と修正を数回
6. Single HTML を生成して納品
